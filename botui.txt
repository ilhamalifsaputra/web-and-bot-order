# PROMPT

Saya ingin memperbaiki UX bot Telegram order ini. Jangan mengubah business logic, payment provider, atau flow backend yang sudah ada. Fokus utamanya adalah memperbaiki flow navigasi agar lebih rapi, konsisten, dan menggunakan satu pesan utama yang diedit terus menerus (editMessageMedia, editMessageCaption, editMessageText, editMessageReplyMarkup), bukan membuat pesan baru setiap langkah.

## Prinsip utama

* Chat user harus bersih.
* Navigasi dilakukan dengan edit message, bukan spam pesan baru.
* User hanya melihat satu "halaman" yang berubah.
* Semua state user harus tersimpan di session.
* Jangan mengubah implementasi payment method yang sudah ada. Gunakan adapter yang ada.
* Jika terdapat beberapa payment method, semuanya harus tetap didukung.

---

# Flow yang diinginkan

## HOME

Saat /start:

Tampilkan informasi user dan keyboard persistent:

```text
Daftar Produk | Saldo

Riwayat Transaksi

Produk Populer | Bantuan
```

Semua tombol menggunakan callback.

Menampilkan produk dengan pagination.

Contoh:

```text

2. ChatGPT
3. CapCut
4. Canva
5. Discord Nitro

Halaman 1/4

Masukkan nomor produk untuk lanjut.
```

Keyboard:

```text
← Sebelumnya | Selanjutnya →
```

User dapat:

* klik next page
* klik previous page
* mengirim nomor produk

Jangan membuat pesan baru.

## WAIT PAYMENT

Setelah payment dibuat:

Tampilkan invoice yang sudah ada saat ini.

Tidak perlu mengubah provider.

Gunakan response yang sekarang.

Tambahkan tombol:

```text
[ Batalkan ]
```

dan jika payment method mendukung refresh:

```text

[ Batalkan ]
```

## CANCEL FLOW

Saat user klik Batalkan:

1. Batalkan invoice jika provider mendukung cancel.
2. Jika provider tidak mendukung cancel, cukup tandai invoice sebagai cancelled.
3. Release stock yang sebelumnya direserve.
4. Hapus pending order.
5. Reset session invoice.
6. Kembalikan state ke PRODUCT_DETAIL.
7. Edit pesan yang sama menjadi halaman PRODUCT DETAIL lagi.
8. Stok terbaru harus tampil.

Tidak boleh mengirim pesan baru.

---

## SUCCESS PAYMENT

Jika pembayaran berhasil:

* tandai order paid
* delivery produk
* update statistik
* reset session
* tampilkan halaman sukses

Keyboard:

```text
[Beli Lagi]

[Daftar Produk]

[Riwayat]
```

# Order-bot customer UX refactor — single-bubble navigation

## Context

`botui.txt` asks for the customer-facing Telegram flow (`apps/order-bot`) to behave
like one continuously-edited message per chat instead of spamming new bubbles per
step, with payment methods sourced dynamically (never hardcoded) and explicit
state. The user supplied a reference bot ("Premiumisme") screenshot set that shows
exactly this: one banner photo whose caption is edited across Home → Product List
→ Product Detail → Order Summary → Payment Wait → Cancel-back-to-Detail, pure
inline keyboards throughout, plus a sold-count line, a "Produk Populer" entry, and
a Refresh button on Product Detail.

The user then supplied screenshots of **this repo's own bot** reproducing the bug:
Product List renders as a banner bubble, but selecting a product or hitting Back
spawns a brand-new message instead of editing in place. Root cause (confirmed by
reading the code): `apps/order-bot/src/util/chat.ts`'s `smartEdit`/`renderMenu`
only edit in place when the update is a callback query **and** the attached markup
is an `InlineKeyboard`. Home (`startCommand`/`showMainMenu`) and Product List
(`browseProductsFlat`/`handleProductNumber`) render with `ckb.mainPersistentKb()` /
`ckb.productsPersistentKb()` — grammY `Keyboard` (reply keyboard) objects — so
every transition touching those two screens forces a fresh send. Everything past
them (denomination picker/detail, order confirmation, payment instructions,
payment wait) already uses real `InlineKeyboard`s and already edits in place
correctly. There is dead code for exactly the inline replacement already sitting
in `apps/order-bot/src/keyboards/customer.ts`: `mainMenu()` and `productsPageKb()`
— defined, never called — evidence this was inline before and got swapped to a
reply keyboard at some point.

Decisions locked in with the user:
1. Add a literal `BotState` enum + explicit session fields (state, page,
   productId, variantId, quantity, paymentMethod, invoiceId, orderId,
   lastMessageId) per `botui.txt`'s spec, layered onto the existing `SessionData`/
   scratch shape — drop any field that ends up genuinely redundant once wired
   through, as a cleanup pass at the end (user's own words: "pake 2, nanti clean
   saja pas selesai kalo ada yg ga dipake").
2. Referral, Language, FAQ, Terms, Support, My Tickets (features the reference
   bot's minimal Home doesn't show but this bot already has) move into a new
   **"Pusat Bantuan" hub screen** off Home — not removed.
3. Extend the existing live-edit-to-success mechanism (already used by Binance
   Internal / Bybit) to TokoPay/QRIS, **without** giving the storefront web route
   direct access to `bot.api` — CLAUDE.md hard rule: "Never send Telegram from the
   web." The mechanism must stay bot-side.

Non-goals: no change to payment-provider logic, stock/order business rules, or DB
schema beyond what's strictly needed (the `Order.paymentMsgChatId`/`paymentMsgId`
columns already exist and are generic — no migration needed). `pnpm typecheck` and
`pnpm test` must stay green throughout; `packages/core/locales/en.json` and
`id.json` must keep identical key sets (enforced by `packages/core/src/locales.test.ts`).
Do not touch the unrelated in-progress worktree at `.worktrees/payment-gateways`
(branch `feature/nowpayments-paydisini`) — just keep the "don't hardcode payment
methods" pattern compatible with it gaining new methods later.

## 1. Session / state model

`apps/order-bot/src/context.ts`: add

```ts
export enum BotState {
  HOME, PRODUCT_LIST, PRODUCT_DETAIL, ORDER_SUMMARY,
  WAIT_PAYMENT, PAYMENT_SUCCESS, HISTORY, HELP, BALANCE,
}
```

and extend `SessionData` with `state: BotState` (default `HOME` in
`initialSession()`) plus the spec's literal fields, mapped onto what already
exists rather than duplicated:
- `page` — alias of the existing `scratch.browsePage` (keep `browsePage` as the
  real storage the existing browse code reads/writes; `page` becomes a thin
  getter/setter pair in the same module, or just rename `browsePage` → `page` at
  the few call sites in `customer.ts` — rename is cleaner than aliasing).
- `productId` / `variantId` — `productId` maps to the existing `viewingProductId`
  scratch field (mid-tier Product); `variantId` maps to `viewingDenomId` (the SKU,
  called "denomination" in this codebase but "variant" in `botui.txt`'s wording).
  Rename the scratch fields to match rather than keep two names for the same
  thing.
- `quantity` — currently threaded through callback_data (`v1:buy:<id>:<qty>`)
  rather than session; ADD it to session too so `BotState`-driven re-renders (e.g.
  Refresh) don't need the qty round-tripped through every callback_data. Keep
  callback_data carrying qty as well (cheap, avoids a stale-session edge case if
  two tabs/devices are open) — session value is the source of truth for redraws
  triggered without a fresh callback (e.g. the cancel-flow's return-to-detail).
- `paymentMethod`, `invoiceId` → set when `buyNowInternal`/`buyNowBybit`/
  `buyNowTokopay`/`buyNow` create an order; `invoiceId` is the order's
  `paymentRef` (or `orderCode` when a method has no separate ref, e.g. Bybit).
- `orderId` — already implicitly known via callback_data everywhere; add to
  session so the WAIT_PAYMENT/PAYMENT_SUCCESS screens and the new Refresh-Status
  button don't need it passed through every call.
- `lastMessageId` — this is exactly the existing `menuMsgId`. Do not duplicate;
  keep `menuMsgId` as the real field (it's load-bearing in `chat.ts`, `checkout.ts`,
  the timers module, and every poller's anchor logic) and treat
  `lastMessageId`/`menuMsgId` as the same field under its existing name. (This is
  the first concrete candidate for "clean up if unused" — `botui.txt`'s name never
  needs to exist as a second field.)

Every screen-rendering function sets `ctx.session.state = BotState.X` as the
first or last statement (cheap, makes the state inspectable/loggable, and is what
the user explicitly asked for) but **rendering itself stays driven by which
handler function ran**, not by a switch over `state` — there is no value in
replacing the existing clean callback-router dispatch (`callbacks.ts`) with a
state-switch; `state` is bookkeeping/visibility, not the control-flow mechanism.
This keeps the change additive and low-risk while still satisfying the literal
ask.

  2. A new `sweepDeliveredAwaitingEdit(api)` function, called once per `pollOnce`
     cycle (after the pending-orders loop), that queries
     `listDeliveredTokopayOrdersAwaitingEdit`, calls `onDelivered` for each, then
     `clearOrderPaymentMessage`. This is what catches orders the **webhook**
     delivered (the common/fast path) — the storefront route
     (`apps/storefront/src/routes/checkout.ts`) is untouched and never imports
     `bot.api`; the edit happens up to one `POLL_INTERVAL_SECONDS` later
     (default 10s) from the existing bot-side poller, fully respecting "never send
     Telegram from the web." The credential DM (`ORDER_DELIVERED_DM` via outbox)
     still arrives instantly via the existing path — only the visual bubble-flip
     has this small added latency.
- `startPolling`/`pollOnce` signatures unchanged (still just `(api: Api)`), so
  `apps/server/src/index.ts`'s wiring needs no changes.
